# Meal Planet: Technical Specification & SRA (Revision 4 — Reconciled)

## 1. Project Overview

A single-family application for automating meal planning, grocery aggregation, and budget estimation. Deployed via Docker. Features a toggle-based serving size configuration — "Boy Weeks" (4 servings) and "Girls Weeks" (6 servings) — to dynamically scale requirements.

**Deployment**: Docker container (docker-compose), local/self-hosted. No public hosting required.
**Auth**: Single-family, no user registration. One family record seeded at init.

---

## 2. Software Requirements Analysis (SRA)

### 2.1 Functional Requirements

- **FR1: Recipe Management** — CRUD for recipes, instructions, and metadata.
- **FR2: LLM-Powered Recipe Parsing** — Automated extraction via Gemini 2.5 Flash. Processes raw HTML/text into a structured JSON schema, handling unit normalization (e.g., "half a cup" ? 0.5) and generating optimized `walmart_search_term` values.
- **FR3: Dynamic Serving Toggles** — Weekly configuration to switch between Boy/Girls weeks.
- **FR4: Ingredient Normalization** — Scaling logic:

$$Q_{calc} = Q_{base} \times \frac{S_{selected}}{S_{base}}$$

Where $S_{selected} \in \{4, 6\}$.

- **FR5: Unified Scheduler** — Calendar to assign recipes and flag week types.
- **FR6: Grocery Aggregation** — Deduplicated list with unit conversion and print/text export.
- **FR7: Walmart Pricing** — Playwright worker using LLM-generated search terms to map items and estimate costs.
- **FR8: Dinner Spinner** — Weighted selection: >14 days (3x), 7–14 days (2x), <7 days (1x).

### 2.2 Non-Functional Requirements

- **NFR1: Async Processing** — Dramatiq + Redis for LLM parsing and Walmart scraping.
- **NFR2: Interoperability** — .ics calendar exports.
- **NFR3: FOSS Context** — Using open-source SDKs/drivers; Gemini API usage is covered by existing personal credits.

## 4. Reconciled Data Schema (PostgreSQL)

```sql
-- Core Family Structure
CREATE TABLE families (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name VARCHAR(255) NOT NULL
);

-- Normalized Ingredients
CREATE TABLE ingredients (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name VARCHAR(255) UNIQUE NOT NULL,
  category VARCHAR(50)
);

-- Recipe Definition (Base Servings)
CREATE TABLE recipes (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  family_id UUID REFERENCES families(id),
  title VARCHAR(255) NOT NULL,
  instructions TEXT,
  source_url TEXT,
  base_servings INTEGER DEFAULT 4,
  last_cooked_at TIMESTAMP,
  created_at TIMESTAMP DEFAULT NOW()
);

-- Junction: Recipes to Ingredients (Updated)
CREATE TABLE recipe_ingredients (
  recipe_id UUID REFERENCES recipes(id) ON DELETE CASCADE,
  ingredient_id UUID REFERENCES ingredients(id),
  quantity DECIMAL NOT NULL,
  unit VARCHAR(50) NOT NULL,
  -- Generated by Gemini to improve scraper accuracy
  walmart_search_term VARCHAR(255),
  PRIMARY KEY (recipe_id, ingredient_id)
);

-- Weekly Configuration (The "Week Type" Toggle)
CREATE TABLE week_configs (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  family_id UUID REFERENCES families(id),
  week_start_date DATE UNIQUE NOT NULL,
  week_type VARCHAR(20) CHECK (week_type IN ('Boy Week', 'Girls Week')),
  serving_override INTEGER NOT NULL
);

-- Scheduling
CREATE TABLE meal_plan (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  family_id UUID REFERENCES families(id),
  recipe_id UUID REFERENCES recipes(id),
  planned_date DATE NOT NULL,
  meal_type VARCHAR(20)
);

-- Retail Store Mappings
CREATE TABLE store_mappings (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  ingredient_id UUID REFERENCES ingredients(id),
  store_item_id VARCHAR(255),
  last_price DECIMAL(10, 2),
  updated_at TIMESTAMP DEFAULT NOW()
);

-- Tags for recipe categorization (Spinner filtering)
CREATE TABLE tags (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name VARCHAR(100) UNIQUE NOT NULL
);

-- Junction: Recipes to Tags (many-to-many)
CREATE TABLE recipe_tags (
  recipe_id UUID REFERENCES recipes(id) ON DELETE CASCADE,
  tag_id UUID REFERENCES tags(id) ON DELETE CASCADE,
  PRIMARY KEY (recipe_id, tag_id)
);

-- Background Job Queue
CREATE TABLE job_queue (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  job_type VARCHAR(50) NOT NULL,
  status VARCHAR(20) DEFAULT 'pending' NOT NULL,
  payload TEXT,
  result TEXT,
  created_at TIMESTAMP DEFAULT NOW(),
  updated_at TIMESTAMP DEFAULT NOW()
);
```

## 5. Implementation Logic (The Gemini Pipeline)

### 5.1 One-Shot Extraction Job

When a URL is submitted, the worker container triggers a Dramatiq actor:

1. **Fetch**: httpx grabs the raw HTML body.
2. **Parse**: Send text to Gemini with a System Instruction to output JSON.
3. **Schema Enforcement**: Use Pydantic to validate the Gemini output before inserting into PostgreSQL.

### 5.2 Optimized Walmart Mapping

Instead of fuzzy-matching ingredient names in Python, the worker uses the LLM's suggested `walmart_search_term` (e.g., "Great Value Garlic Cloves 3-pack"). This bypasses the need for complex internal cleaning logic.

### 5.3 Serving Size Scaling Logic

The aggregation service performs a look-up for the `week_type` associated with the `planned_date` in the `meal_plan`.

```python
def calculate_required_quantity(recipe_ingredient, target_serving):
    scaling_factor = target_serving / recipe_ingredient.recipe.base_servings
    return recipe_ingredient.quantity * scaling_factor
```

### 5.4 The "Dinner Spinner" Weighting

To avoid redundancy, the spinner uses the `last_cooked_at` column:

- **>14 days** (or never cooked): weight **3x**
- **7-14 days**: weight **2x**
- **<7 days**: weight **1x**
